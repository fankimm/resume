Technical Highlight

문제 해결 과정과 결과

Problem

배송확정 페이지에서 5만건 이상 주문 데이터 조회 시 57초 소요. 물류 운영팀의 업무 병목 발생.

Solution

프론트 쪽 중복 API 호출 제거, 가상 스크롤 적용, 페이지네이션 전환. 백엔드와 협업하여 API 구조 재설계.

Result

조회 시간 57초 → 5초로 단축. 프론트·백엔드 양쪽 개선의 결과.

Problem

3PL 물류 확장으로 4개 화주사 동시 운영 필요. 화주별 비즈니스 로직과 API가 모두 다름.

Solution

화주별 API 라우팅 분기 설계 및 공통 컴포넌트 추상화. 브랜드 독립적인 모듈 구조 구축.

Result

4개 화주사(PAF, WAAC, BOSE, BNW)를 단일 시스템에서 안정적으로 운영.

MH Save Editor

Reverse Eng.

2017 · Personal

개발자가 되기 전, 순수한 호기심으로 만든 첫 프로그램. 몬스터헌터 세이브 파일을 헥스 에디터로 분석하여 바이너리 데이터 구조를 역공학하고 C/C++로 아이템 에디터를 구현했다.

이 프로젝트를 만들 때는 Git이 뭔지도 몰랐다. 당연히 소스 코드는 유실되었고, 남은 건 컴파일된 바이너리 파일 하나뿐. 이력서를 작업하면서 스크린샷이라도 넣고 싶었지만 세이브 파일이 없어 실행조차 할 수 없었다. 결국 남은 바이너리에서 strings와 otool로 UI 문자열과 구조를 추출한 뒤, AI를 활용하여 소스 코드를 복원했다.

주요 작업

  • 처음에는 아이템 데이터가 바이트 단위일 거라 생각하고 접근 — 첫 번째 슬롯 수정은 성공했지만, 두 번째 슬롯을 건드리면 첫 번째까지 깨지는 현상 발생
  • 1주일간의 삽질 끝에 데이터가 바이트가 아닌 비트 단위로 패킹되어 있다는 것을 발견 — 아이템 ID 12비트 + 수량 7비트 = 슬롯당 19비트, 바이트 경계를 무시하는 구조
  • 비트 단위 읽기/쓰기 로직을 구현하여 CLI 에디터 완성
  • 소스 유실 후 바이너리 분석 + AI 활용으로 소스 코드 복원 (2026)
